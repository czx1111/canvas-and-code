## 聪明的诱惑

每个开发者都会经历一个阶段，认为好代码就是*聪明*的代码。抽象工厂、深层继承层次、元编程、配置层上叠配置层。

这个阶段通常持续大约三年。然后某个东西坏了，你花了一周去调试，然后你意识到*聪明是一种负债*。

## 简洁 > 聪明

我读过的最好的代码是无聊的。它是一系列直截了当的步骤，每一步都显而易见，每一步都能独立阅读。没有惊喜。没有"啊哈"时刻。只是……清晰。

这里有一个测试：如果一个初级开发者在读你的代码时不能不问你问题就理解它，那它太聪明了。

## 三个原则

### 1. 扁平胜于嵌套

```python
# 差：五层嵌套
def process(data):
    if data:
        for item in data:
            if item.active:
                if item.type == "A":
                    # ... 做点什么

# 好：扁平，提前返回
def process(data):
    if not data:
        return
    for item in data:
        if not item.active:
            continue
        if item.type == "A":
            # ... 做点什么
```

### 2. 显式胜于隐式

魔法在小说里很棒。在代码里，它很可怕。当你读一行代码时，你应该准确知道它做什么，而不需要查阅另外三个文件。

### 3. 删除是最好的重构

最好的优化是删除你不需要的代码。每一行代码都是必须维护、理解并最终被替换的行。行数越少越好。

## 真正的复杂性

关键不是代码应该简单化。真正的问题有真正的复杂性。关键是我们应该*把复杂性预算放在重要的地方*——在问题域，而不是管道中。

一个博客不需要插件系统。一个待办应用不需要微服务架构。一个落地页不需要状态管理库。

> 完美不是无可增加，而是无可削减。

## 结论

写简洁代码比写聪明代码更难。它要求你深入理解问题，抵制过早抽象化的冲动，并相信直接的解决方案是足够的。

但这是值得的。因为六个月后，当你回到那段代码时，你会感谢自己。
