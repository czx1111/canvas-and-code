## 问题：传统 I/O 为什么要拷贝两次

传统 `read()` 的数据流向：

```
磁盘 ──DMA拷贝──→ 内核缓冲区(页缓存) ──CPU拷贝──→ 用户进程缓冲区
```

**为什么会这样？** 因为 I/O 操作在**内核态**执行，磁盘数据先被 DMA 搬到内核的页缓存（page cache）。而用户进程在**用户态**运行，无法直接访问内核内存。

所以用户进程拿数据只有两条路：

1. **每次访问都切换到内核态** — 上下文切换开销极大，效率太低
2. **把数据从内核缓冲区拷贝到用户缓冲区** — 传统 `read()` 的做法，多了一次 CPU 拷贝

此外，整个 `read()` 过程涉及 **2 次上下文切换**（用户态→内核态→用户态）+ **2 次拷贝**（DMA + CPU）。

## mmap 的核心思路：不拷贝，而是映射

`mmap()` 的做法是：**把进程虚拟地址空间中的一段区域，直接映射到内核页缓存中对应的物理页**。

```
磁盘 ──DMA拷贝──→ 内核页缓存(物理页)
                      ↑
               页表映射(同一物理页)
                      ↑
              进程虚拟地址空间
```

关键在于：用户进程的虚拟地址和内核的页缓存指向 **同一块物理内存**。不需要任何 CPU 拷贝——进程直接通过指针读取数据，本质上就是在访问内核的页缓存。

## 工作流程

### 1. 建立映射

```c
void *addr = mmap(NULL, len, PROT_READ, MAP_SHARED, fd, 0);
```

此时内核只是**修改页表**，建立了虚拟地址到文件页的映射关系，还没有真正加载数据。

### 2. 按需加载（缺页中断）

当进程第一次访问这段虚拟地址时，触发**缺页中断（page fault）**：

- 内核发现该虚拟页对应的物理页不存在
- 从磁盘加载数据到物理页（通过 DMA）
- 更新页表，将虚拟地址指向该物理页
- 进程恢复执行，直接读到数据

> 这种"用到时才加载"的机制叫**懒加载（lazy loading）**，只加载实际访问的页面，避免一次性读入整个文件。

### 3. 写回磁盘

对于写操作，修改后的页会被标记为**脏页（dirty page）**。内核会在合适时机（或调用 `msync()` 时）将脏页刷回磁盘。

## 传统 read vs mmap 对比

| 对比项 | 传统 `read()` | `mmap()` |
|--------|---------------|-----------|
| CPU 拷贝 | 需要（内核→用户） | **不需要** |
| 上下文切换 | 2 次 | 0 次（首次访问触发 1 次缺页中断） |
| 数据加载 | 一次性读入 | 按页懒加载 |
| 用户访问方式 | 先 `read` 再访问缓冲区 | 直接通过指针访问 |

## 适用场景

- **大文件读取**：不需要一次性读入整个文件，按需加载节省内存
- **多进程共享内存**：映射同一个文件，多个进程通过页表指向同一物理页，天然实现共享
- **零拷贝优化**：配合 `sendfile()` 或 `splice()` 可进一步消除用户态参与

## 注意事项

> mmap 不是万能的。

- **小文件**：建立映射、缺页中断的开销可能比直接 `read()` 还大
- **随机写**：频繁的脏页刷写可能导致性能下降
- **信号风险**：访问映射区域如果出错会触发 `SIGSEGV`，需要做好异常处理
- **映射开销**：建立和释放映射有固定成本，不适合频繁创建销毁的场景

## 本质

内存映射文件的本质是：**用页表代替拷贝**。通过让用户进程的虚拟地址直接指向内核页缓存的物理页，省去了内核到用户空间的数据拷贝，这就是"零拷贝"思想的一种体现。
